🎓 ГАЙД: Підключаємо Claude Code до CRM (Bitrix24, Kommo, KeyCRM, HubSpot, Pipedrive) — від токена до аудиту в Google Drive

Цей гайд для тих, хто вже має VPS з Claude Code і Telegram-бот, підключений до сервера. Якщо ні — спочатку гайди з налаштування VPS і Claude Code.

**Реальний кейс, на якому виріс цей гайд.** Підключила Claude Code до Bitrix24 одного агентства нерухомості з офісом у Дубаї. За **2,5 години вночі** він пройшов 6 400 лідів, 4 141 угоду, 16 689 контактів, 9 воронок, 93 кастомних поля у сделок, посчитал ВСІ виграні угоди по всіх країнах (по Дубаю — 93M AED + 24M USD + 2,4M EUR) і знайшов, що поле «Комісія» заповнене лише у 12% виграних угод. Тобто компанія роками не бачила своєї реальної маржі. Аудит + ТЗ для розробника лежав у Google Drive до ранку.

Раніше така робота — це аналітик на 2-3 дні. Зараз — кілька промтів.

**Що робиш ти:** один раз береш у CRM webhook або API-ключ і даєш Claude Code.
**Що робить Claude Code:** інвентаризує всі сутності, аналізує воронки, шукає дірки в даних, пише ТЗ для розробника, кладе документ у Drive — сам, через промти нижче.

## Що потрібно від тебе (5-10 хвилин у CRM)

Одна дія вручну на кожну CRM — отримати webhook або API-токен. Усе інше робить Claude Code.

### Bitrix24 (вхідний webhook)

1. У кабінеті Bitrix24: **Розробникам** → **Інше** → **Вхідний вебхук**
2. Постав **усі** галочки в розділах `crm`, `user`, `department`, `tasks` (без них падає з `insufficient_scope`)
3. Скопіюй URL виду:
   `https://your-domain.bitrix24.eu/rest/60/abcd1234efgh5678/`
4. Звідси Claude робить будь-який метод REST API: `crm.deal.list`, `crm.userfield.list`, `crm.status.list` тощо.

### Kommo (Long-lived token + subdomain)

1. **Налаштування** → **Інтеграції** → створи приватну інтеграцію
2. Постав права: CRM, Контакти, Угоди, Каталоги
3. Скопіюй **Long-lived token** і свій **subdomain** (`yourcompany.kommo.com`)
4. Claude використовує `https://{subdomain}/api/v4/leads`, `/contacts`, `/companies` з заголовком `Authorization: Bearer {token}`.

### KeyCRM (API key)

1. **Налаштування** → **API** → згенеруй ключ
2. Скопіюй ключ (один рядок)
3. Claude б'є `https://openapi.keycrm.app/v1/order`, `/buyer`, `/product` з заголовком `Authorization: Bearer {key}`.

### HubSpot (Private App token)

1. **Settings** → **Integrations** → **Private Apps** → Create
2. Scopes: `crm.objects.contacts.read`, `crm.objects.deals.read`, `crm.objects.companies.read`, `crm.schemas.deals.read` (мінімум read; додай write якщо хочеш щоб Claude правив дані)
3. Скопіюй токен (починається з `pat-`)
4. Claude використовує `https://api.hubapi.com/crm/v3/objects/deals` з `Authorization: Bearer {token}`.

### Pipedrive (API token)

1. У правому верхньому куті: **Personal preferences** → **API**
2. Скопіюй токен
3. Claude б'є `https://your-domain.pipedrive.com/api/v1/deals?api_token={token}` (токен як параметр, не header).

## Промт 1 — Зберегти токен на сервер у .env

Скопіюй у Claude Code, підстав свій ключ:

```
Збережи токен Bitrix24 на сервер.
URL вебхука: https://my-domain.bitrix24.eu/rest/60/AAAA1234BBBB5678/

Запиши у /home/botuser/claudebot/.env як змінну BITRIX_WEBHOOK_URL
(з косою рискою в кінці). Якщо змінна вже є — заміни значення.
Потім зроби тестовий виклик crm.deal.list з limit=1 і покажи відповідь.
```

Для інших CRM пишеш аналогічно — `KOMMO_TOKEN` + `KOMMO_SUBDOMAIN`, `KEYCRM_TOKEN`, `HUBSPOT_TOKEN`, `PIPEDRIVE_TOKEN` + `PIPEDRIVE_DOMAIN`.

Тестовий виклик потрібен щоб одразу побачити: «scope=insufficient» — і повернутися додати галочки в кабінеті.

## Промт 2 — Повна інвентаризація CRM

Це найважливіший промт. Виконується одноразово і дає Claude карту твоєї CRM. Далі будь-який аналіз спирається на цю карту.

```
Зроби повну інвентаризацію Bitrix24. Збережи результат як JSON файли в репо
(каталог cmds/bitrix/) і запуш у поточну гілку.

Збери:
1. Профіль власника вебхука (user.current) + список активних користувачів
2. Усі типи сутностей CRM (crm.status.entity.types)
3. Усі стадії і статуси (crm.status.list) — розгрупуй по ENTITY_ID
4. Усі воронки сделок (crm.category.list для DEAL) — id + назва
5. Усі кастомні поля сделок (crm.deal.userfield.list) і лідів (crm.lead.userfield.list)
6. Перелік усіх департаментів (department.get)
7. Загальні лічильники: total по лідах, угодах, контактах, компаніях
8. Останні 20 лідів і 20 угод (для прикладу структури)

Збережи 12 JSON-файлів: 00_scope.json, 01_user_current.json, 02_users.json,
03_status_types.json, 04_statuses.json, 05_deal_categories.json,
06_deal_userfields.json, 07_lead_userfields.json, 08_departments.json,
09_totals.json, 10_recent_leads.json, 11_recent_deals.json.

У stdout виведи коротке резюме: скільки сутностей кожного типу,
скільки кастомних полів, скільки воронок, скільки активних користувачів.
```

**Що отримаєш у stdout** (приклад з реального кейсу):

```
=== TOTALS ===
  lead    : total=6400
  deal    : total=4141
  contact : total=16689
  company : total=378

=== CUSTOM FIELDS ===
  Deal UF_: 93
  Lead UF_: 36

=== DEPARTMENTS ===
  Marketing department, Call center/CRM, Team 1, Team 2 ...

=== DEAL CATEGORIES (воронки) ===
  CAT= 0 Bali
  CAT= 6 Dubai
  CAT=34 Baku
  CAT=36 Georgia
  CAT=44 Maldives
  ...
```

Тепер Claude знає твою CRM напам'ять. Можна ставити будь-яке аналітичне питання.

## Промт 3 — Аналіз виграних угод по воронках

```
По кожній воронці порахуй:
- скільки всього угод
- скільки в стадії WON (для кожної воронки знайди свій ID won-стадії — наприклад
  у Bitrix це не завжди WON, може бути C6:UC_0AV0VK)
- суму WON по валютах (поле OPPORTUNITY + CURRENCY_ID)
- топ-15 виграних угод по сумі
- топ-5 менеджерів за обертом WON

Окремо порахуй те ж саме тільки за поточний рік (фільтр по CLOSEDATE або DATE_CREATE).

Виведи табличкою у stdout.
```

Реальний результат (по Дубаю):

```
CAT= 6 Dubai   won=C6:UC_0AV0VK   69 угод | 93 165 253 AED + 2 394 550 EUR + 24 572 097 USD
CAT= 0 Bali    won=UC_9J2SPG       0 угод
CAT=34 Baku    won=C34:UC_CYM0X8   2 угоди | 1 402 275 AED
CAT=36 Georgia won=C36:UC_J3FRNY   1 угода
CAT=44 Maldives won=C44:UC_D7TAS7  1 угода | 1 930 000 AED
```

**Грабля:** won-стадія в Bitrix НЕ завжди `WON`. У кастомних воронках вона часто має ID типу `C6:UC_0AV0VK` — це треба брати з `04_statuses.json` (з інвентаризації) для кожної воронки окремо. Claude робить це сам.

## Промт 4 — Знайти дірки в критичних полях

Це той самий промт, який у реальному кейсі знайшов 88% угод без поля «Комісія».

```
Знайди у кастомних полях сделок ті, що відповідають за гроші і комісію
(тип money, або назва містить commission/комис/profit).

Для кожного такого поля порахуй:
- у скількох виграних угодах воно заповнене
- у скількох порожнє
- % заповненості

Окремо зроби розбивку по роках (по даті закриття) і по воронках.

Виведи табличкою. Якщо % заповненості нижчий 50% — це червоний прапор,
підсвіти.
```

Результат у реальному кейсі:

```
Воронка       Всього WON   З комісією    %
Dubai              69            7   10.1%   ← червоний прапор
Bali                0            0    —
Baku                2            1   50.0%
Georgia             1            1  100.0%
Maldives            1            0    0.0%   ← червоний прапор

ИТОГО WON: 73 | з комісією: 9 (12.3%)
```

12.3% — це означає, що власник бізнесу **в принципі не бачить реальної маржі** по 88% виграних угод. Це не питання дисципліни брокерів — це питання обов'язковості поля в CRM-процесі. Те, що знайшов Claude за 10 хвилин, всередині компанії ніхто не помічав роками.

Цей же промт працює на будь-якій іншій CRM — назви полів будуть інші, логіка та сама.

## Промт 5 — Підготувати ТЗ для розробника + завантажити в Google Drive

```
На основі попереднього аналізу зроби коротке ТЗ (Markdown) для нашого
розробника-Insider'а Bitrix24. ТЗ має містити:

1. Поточну ситуацію (факти з аналізу — кількість, %, валюти)
2. Гіпотезу чому так (наприклад: поле «Комісія» не обов'язкове)
3. Перелік змін у CRM:
   - які поля зробити обов'язковими (на якій стадії)
   - які стадії додати/прибрати у воронці
   - які автоматичні перевірки додати (наприклад блокувати перехід у WON
     поки порожня Комісія)
4. Перелік звітів, які треба зробити стандартними (Дашборд для власника)
5. Орієнтовний обсяг робіт (з твоєї експертизи)

Збережи файл як BITRIX_audit_YYYY-MM-DD.md у репо (dominanta-artifacts/).

Потім завантаж його як Google Doc у папку CRM в Drive клієнта.
ID папки в Drive: 1whpRd926oDcvQbahf_lW2FgTOGLK4lYj
(якщо немає її — створи в корені).

Назва документа: "CRM. Аудит [НАЗВА_CRM] і ТЗ для Insider (DD.MM.YYYY)"

Поверни мені посилання на створений Google Doc.
```

Claude використає `oauth2.googleapis.com/token` з твоїм `GOOGLE_OAUTH_REFRESH_TOKEN` і `drive.files.create` з `application/vnd.google-apps.document`. Якщо ще не підключав Drive — спочатку гайд по підключенню Google API.

## Граблі з реального досвіду

| Грабля | Що робити |
|---|---|
| `insufficient_scope` при першому виклику | Перестворити webhook з УСІМА галочками `crm`, `user`, `department` |
| Won-стадія не дорівнює `WON` | Брати ID won-стадії з `04_statuses.json` окремо для кожної воронки |
| Сделки в кількох валютах одночасно | Не сумуй конвертацією, агрегуй по валютах окремо (AED, USD, EUR) |
| Дублі: один клієнт у різних воронках | Шукай по нормалізованому імені/email/телефону, попроси Claude витягти таких |
| 93 кастомних поля з технічними назвами `UF_CRM_1714411870` | Claude витягує `LABEL` з `crm.deal.userfield.list` — назви українські/російські стають читабельними |
| Pagination: REST CRM повертає по 50 записів | Claude автоматично робить пагінацію через `start` параметр |
| `OPPORTUNITY` зберігається як рядок | Claude конвертує в число перед сумуванням |
| Воронка CAT=0 = «За замовчуванням» — не означає що там нічого | У реальному кейсі там було 154 угоди |

## Що ти робиш руками (мінімум)

| Дія | Чому не можна автоматизувати |
|---|---|
| Створити webhook/токен у CRM | Потребує входу в кабінет з 2FA |
| Виставити правильні scope при створенні | Один раз, відповідальне рішення |
| Підтвердити дозвіл на Google Drive | OAuth flow з браузером |
| Дивитися на цифри і робити висновки | Це **твоя** експертиза як власника процесу |

**Claude Code робить усе інше:** будь-які запити в API, агрегацію, аналітику, пошук дублів, перевірку заповненості, генерацію ТЗ, завантаження в Drive, оформлення документа, надсилання звіту в Telegram.

## Адаптація під інші CRM

Цей самий патерн працює для **Kommo, KeyCRM, HubSpot, Pipedrive** — змінюються тільки URL і назви методів. Логіка інвентаризації, агрегації і ТЗ — однакова.

| CRM | Базовий endpoint | Метод "перелік угод" |
|---|---|---|
| Bitrix24 | `{webhook}/` | `crm.deal.list` |
| Kommo | `https://{sub}.kommo.com/api/v4/` | `leads` |
| KeyCRM | `https://openapi.keycrm.app/v1/` | `order` |
| HubSpot | `https://api.hubapi.com/crm/v3/objects/` | `deals` |
| Pipedrive | `https://{domain}.pipedrive.com/api/v1/` | `deals?api_token=...` |

Просто скажи Claude у промті 1 яка CRM — далі він пише запити сам.
