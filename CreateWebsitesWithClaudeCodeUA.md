## Що потрібно від тебе (10 хвилин)

Три одноразові дії — далі все автоматично через Claude.

### 1. GitHub-токен (якщо ще немає)

1. Зайди на [github.com/settings/tokens](https://github.com/settings/tokens) → "Generate new token (classic)"
2. Назва: `claude-websites`, scope: ☑ `repo`, без терміну
3. Скопіюй токен — починається з `ghp_...`

### 2. Cloudflare Global API Key

1. Зайди на [dash.cloudflare.com/profile/api-tokens](https://dash.cloudflare.com/profile/api-tokens)
2. Прокрути вниз до секції **API Keys**
3. Напроти **Global API Key** натисни **View** → введи пароль → скопіюй

⚠️ Це майстер-ключ з повним доступом до Cloudflare. Зберігається на сервері з chmod 600 і ніколи не коміт у git. Якщо засвітиться — на тій самій сторінці кнопка **Roll** генерує новий, старий миттєво відкликається.

### 3. Підключити Lovable до GitHub (якщо сайт у Lovable)

В Lovable: відкрий проєкт → шапка справа → іконка GitHub → **Connect to GitHub** → погодься. Lovable створить приватне репо і запише туди весь код.

Все. Далі працює Claude Code.

## Промт 1 — Зберегти токени на сервер

Скопіюй цей текст у Claude Code, підстав свої значення:

```
Збережи на сервер у /home/botuser/claudebot/.env наступні токени
через cmd_runner (ніколи не коміть у git!):

GITHUB_TOKEN=ghp_твій_токен
CLOUDFLARE_GLOBAL_KEY=cfk_твій_ключ
CLOUDFLARE_EMAIL=твій-email@gmail.com
CLOUDFLARE_ACCOUNT_ID=

(CLOUDFLARE_ACCOUNT_ID отримай через API і одразу запиши.)

Перевір що дублів немає, chmod 600 на .env.
```

**Що зробить Claude Code:** запише токени в `.env` на сервері, через виклик API Cloudflare отримає Account ID і збереже його, перевірить що ключі робочі — поверне «success=true».

## Промт 2 — Клонувати репо і подивитись що там

```
Клонуй мій репо https://github.com/<мій-нікнейм>/<назва-репо>.git
у /home/botuser/build/site/ на сервер. Гілка claude/setup-vercel
(або main якщо такої немає).

Після клонування покажи:
1. Який стек (Next.js / Vite / щось інше)
2. Список усіх сторінок у src/pages
3. Чи є .env.production з публічними ключами (Stripe pk_, Supabase anon)
4. Чи є supabase/functions (бекенд) або api/ (serverless)
```

**Що зробить Claude Code:** через cmd_runner з GITHUB_TOKEN склонує репо на сервер, прочитає файли, поверне список і структуру. Якщо це Lovable-проєкт — побачиш типовий Vite + React + Tailwind + shadcn/ui + Supabase.

## Промт 3 — Деплой на Cloudflare Pages

```
Створи проєкт `<назва-проєкту>` на Cloudflare Pages і задеплой код
з /home/botuser/build/site/ туди.

Кроки:
1. POST /accounts/{ACC}/pages/projects з name=<назва>, production_branch=main
2. На сервері: npm install --no-audit, npm run build
3. Додай у dist/_redirects рядок «/* /index.html 200» (для SPA-роутингу)
4. Деплой через npx wrangler@3 pages deploy dist --project-name=<назва>
   (Wrangler latest потребує Node 22+, а на сервері 20 — тому wrangler@3)
5. Поверни мені preview URL виду https://<хеш>.<назва>.pages.dev
6. Перевір curl-ом що / і кілька сторінок повертають 200
```

**Що зробить Claude Code:** створить проєкт на Pages через API, склонує + збере код на сервері, заллє через Wrangler, поверне preview-URL з підтвердженням що всі сторінки рендеряться.

## Промт 4 — Привʼязати власний домен

```
Привʼяжи мій домен <мій-домен.com> до проєкту Cloudflare Pages «<назва>».

Кроки:
1. POST /accounts/{ACC}/pages/projects/<назва>/domains з name=<мій-домен.com>
2. Знайди zone_id мого домена через GET /zones?name=<мій-домен.com>
3. Якщо у DNS є стара A-запис на чужого хостера — видали її
4. Створи CNAME apex: name=<мій-домен.com>, content=<назва>.pages.dev,
   proxied=true, ttl=Auto
5. Перевір через X-Auth-Email + X-Auth-Key (це Global API Key, не Bearer)
6. Зачекай 30 секунд, потім зроби curl https://<мій-домен.com> і перевір
   що відповідає Cloudflare (server: cloudflare у заголовках)
```

**Що зробить Claude Code:** через Cloudflare API додасть домен у Pages, знайде зону, прибере конфлікти, створить новий CNAME, дочекається SSL і поверне «✅ домен живий».

## Промт 5 — Перебудувати лендінг за структурою конкурентів

Це головний промт коли треба підвищити конверсію існуючого сайту.

```
Я хочу перебудувати лендінг src/pages/<сторінка>.tsx за прикладом конкурентів.

Конкуренти у моїй ніші:
- https://<конкурент-1>
- https://<конкурент-2>
- https://<конкурент-3>

Дій так:
1. Витягни сторінки конкурентів через r.jina.ai (вони здебільшого
   за Cloudflare-челенджем, тому curl напряму не працює)
2. Проаналізуй структуру: які блоки (hero, symptoms, why-method,
   doctor, curriculum, pricing, testimonials, FAQ, final CTA),
   яка довжина курсу, скільки уроків, які заголовки
3. Перебудуй мою сторінку щоб структура збігалася з найкращим конкурентом:
   - Якщо у них «X-Day Method» — постав таку ж
   - Якщо у них N video lessons — постав N
   - Додай Symptoms checklist (8-10 конкретних симптомів)
   - Додай блок «Why This Method Works» (root-cause framing)
   - Зменш «вода-секції» (на кшталт абстрактного Solution з 3 features)
4. Тримай блок «About Doctor» компактно — не розтягуй на пів сторінки
5. Зроби лендінг щільним: 5-6 секцій замість 9-10
6. Закоміть у репо на новій гілці feat/<що-зробив> і деплой preview
7. Поверни preview URL — я подивлюсь і скажу мерджити чи допилити
```

**Що зробить Claude Code:** скрейпить конкурентів через r.jina.ai, складе порівняльну таблицю, перепише сторінку з мерджем секцій, задеплоїть на preview і дасть тобі URL до мерджу в прод.

**Що робиш ти:** дивишся preview, кажеш «ок» або «переробити X». Це єдина точка прийняття дизайнерських рішень.

## Промт 6 — Створити нову сторінку за шаблоном існуючої

Коли є робочий лендінг (наприклад `/back`) і треба зробити схожий під іншу тему (наприклад `/sleep`).

```
Скопіюй структуру моєї робочої сторінки src/pages/Back.tsx у нову
сторінку src/pages/Sleep.tsx.

Тема: «21-Day Sleep Reset» (бессоння + хронічні болі вночі).
Спікер: той самий — Dr. Smirnov.

Адаптуй:
1. Symptoms checklist — 10 симптомів конкретно про сон
   (waking up at 3 am, racing thoughts, neck pain disrupting sleep, etc.)
2. Root causes — 4 причини проблем зі сном через тіло
3. About Doctor — той самий блок, переформулюй цитату під сон
4. Curriculum — 21 урок розбий на 3 тижні: Reset → Unlock → Strengthen
5. Testimonials — 3 нових відгуки, орієнтовані на сон
6. Pricing — той самий $19, value $166
7. FAQ — 5 питань про сон, не про спину

Додай маршрут у App.tsx: <Route path="/sleep" element={<Sleep />} />.
Закоміть, задеплой preview, дай мені посилання.
```

**Що зробить Claude Code:** клонує файл, перепише всі тексти і масиви під сон, додасть маршрут, задеплоїть і поверне URL для перевірки.

## Скіли — встановити один раз для якості сайту

Це файли з експертизою. Claude Code автоматично використовує їх коли ти просиш зробити щось пов'язане.

### Скіл 1: web-design-guidelines (рев'ю UI/UX)

```
Встанови скіл web-design-guidelines з:
https://raw.githubusercontent.com/anthropics/skills/main/skills/web-design-guidelines/SKILL.md
у ~/.claude/skills/web-design-guidelines/SKILL.md
```

Що дає: коли просиш «Зроби рев'ю мого UI», Claude перевіряє accessibility, contrast, typography, spacing — за стандартами Apple/Vercel/Stripe.

### Скіл 2: frontend-design (створення дизайну)

```
Встанови скіл frontend-design з vercel-labs/agent-skills.
```

Що дає: Claude генерує лендінги і компоненти у production-grade стилі, без AI-естетики «однакових сайтів».

### Скіл 3: page-cro (conversion optimization)

```
Встанови скіл page-cro для оптимізації конверсії лендінгів.
```

Що дає: Claude автоматично пропонує CRO-покращення (кращі CTA, social proof, скорочення друк-форм, A/B-ідеї) коли працюєш над лендінгом.

### Скіл 4: copywriting (маркетингові тексти)

```
Встанови скіл copywriting для маркетингових текстів.
```

Що дає: коли просиш «перепиши hero, щоб краще продавав» — Claude застосовує формули PAS, AIDA, JTBD замість загальних слів.
