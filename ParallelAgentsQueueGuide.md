🎓 ГАЙД: Кілька агентів Claude одночасно на одному сервері — паралельна черга команд

Цей гайд для тих, хто вже має VPS із Claude Code і працюючий `cmd_runner`
(якщо ні — спочатку гайд «Перетворюємо Telegram-бота на повноцінного
агента»). Тут ми приберемо біль, через яку всі рано чи пізно проходять.

**Знайома ситуація?** Ти відкрила дві-три сесії Claude, дала кожній свою
задачу — а вони починають сваритися: «команда вже виконується на сервері»,
відповіді плутаються, одна сесія перетирає іншу. У підсумку доводиться
робити все по черзі, по одній. Повільно й дратує.

**Чому так.** Класичний `cmd_runner` має **один слот** на команду: усі сесії
пишуть в один файл `pending.json`, а сервер бере звідти одну команду за раз.
Дві сесії одночасно → друга затирає першу. Це не сервер слабкий — це вузьке
горло в самій конструкції черги.

**Що стане можливим після гайду:**
- кілька агентів (і ти сама в кількох вкладках) працюють **одночасно**;
- до 10 команд виконуються паралельно, решта чекає у черзі — нічого не
  губиться;
- сесії не заважають одна одній і не плутають відповіді.

---

## Ідея простими словами

Замість **однієї поштової скриньки на всіх** (куди всі кидають листи й
перетирають один одного) робимо **окрему скриньку для кожного листа**.

- Раніше: усі пишуть в один файл `pending.json` → бійка за слот.
- Тепер: кожна задача — **окремий файл** у папці `cmds/jobs/`, а відповідь —
  окремий файл у `cmds/results/`. Імена різні → зіткнень немає в принципі.

Сервер просто дивиться, які нові файли з'явились у `cmds/jobs/`, і запускає
їх **паралельно** (пулом воркерів). Виконав — поклав відповідь у
`cmds/results/` і прибрав файл задачі.

---

## Крок 1. Навчи `cmd_runner` читати папку задач

У циклі рунера замість «прочитати один `pending.json`» робимо «переглянути
всі файли в `cmds/jobs/` і кожен новий віддати в пул потоків». Ключові
шматки (Python):

```python
from concurrent.futures import ThreadPoolExecutor
executor = ThreadPoolExecutor(max_workers=10)   # до 10 одночасно
submitted = set()                                # щоб не запускати двічі

# у головному циклі, раз на ~5 секунд:
for entry in list_dir("cmds/jobs"):              # список файлів у папці
    name = entry["name"]
    if not name.endswith(".json") or name in submitted:
        continue
    submitted.add(name)
    job = read_json(entry["path"])               # {id, cmd, cwd?, timeout?}
    executor.submit(run_job, name, job)          # запуск паралельно
```

А `run_job` виконує команду, пише відповідь у `cmds/results/<id>.json` і
видаляє файл задачі з `cmds/jobs/`.

> Порада: записи у Git (коміти відповідей) обгорни одним «замком» (Lock),
> щоб паралельні воркери не штовхалися за гілку. Самі команди при цьому
> виконуються паралельно — а це найповільніша частина.

## Крок 2. Як тепер ставити задачу

Кладеш у `cmds/jobs/` файл із **унікальним** іменем:

`cmds/jobs/moja-zadacha-20260613-0715.json`
```json
{"id":"moja-zadacha-20260613-0715","cmd":"echo привіт із сервера","cwd":"/home/<user>/<bot>"}
```

| Поле | Обов'язкове | Що це |
|---|---|---|
| `id` | так | Унікальний ідентифікатор. Зручно: `назва-дата-час`. |
| `cmd` | так | Команда для сервера. |
| `cwd` | ні | Робоча папка. |
| `timeout` | ні | Скільки секунд чекати (за замовч. 120). |

Запушити у свою гілку. Якщо push відлетів (хтось запушив раніше) —
`git pull --rebase` і повторити. **Файли в усіх різні, тож конфліктів немає.**

## Крок 3. Забрати відповідь

Через ~5–10 секунд читаєш **свій** файл `cmds/results/<твій-id>.json`:

```json
{"id":"moja-zadacha-20260613-0715","stdout":"привіт із сервера\n","returncode":0}
```

Головне правило: **читай саме свій `id`**, не «останню відповідь» — поряд
можуть працювати інші.

---

## Підводні камені (щоб не наступити)

- **`id` завжди унікальний.** Інакше відповіді плутаються.
- **Не вставляй гігантські base64-блоки однією командою.** Це і повільно, і
  система захисту Claude може заблокувати запит («Request was blocked»).
  Треба залити файл на сервер — хай агент **завантажить його через API
  (raw)** прямо у скрипті, а не вставляє величезний base64.
- **Не друкуй секрети** (токени, паролі) у командах — відповідь зберігається
  в Git. Жодних «покажи мені вміст .env».
- **Черга вирішує бійку за канал, але не за дію.** Дві сесії не повинні
  одночасно робити конфліктну дію — наприклад, перезапускати бота чи
  редагувати один і той самий файл. Це вже про логіку задач.

---

## Бонус: хай черга оновлює себе сама

Додай у рунер крихітну перевірку: раз на ~хвилину (коли немає задач у
роботі) він порівнює свій код із гілкою і, якщо змінився, перезапускається з
новим кодом. Тоді **будь-яке оновлення черги ти просто пушиш у гілку** — і
нічого руками на сервері робити не треба.

---

Підсумок: один файл на задачу замість спільного слота + пул воркерів —
і твої агенти працюють пліч-о-пліч, а не по черзі. Спробуй відкрити дві
сесії і дати їм різні задачі одночасно: тепер обидві поїдуть разом. 🚀
